# Безопасность

Описание текущей защиты доступа (Docker + nginx), учётных данных по умолчанию, смена пароля и перечень возможных улучшений.

---

## Текущая защита (Docker)

При запуске через **Docker** весь трафик к приложению идёт через **nginx** (reverse proxy). Снаружи открыт только порт **8080**; контейнер с приложением (Go) снаружи не доступен.

### Учётные данные по умолчанию

| Поле     | Значение |
|----------|----------|
| **Логин**  | `admin` |
| **Пароль** | `admin` |

Их нужно вводить при первом запросе к http://localhost:8080 (страница, API, WebSocket). **Для продакшена обязательно смени пароль** (см. ниже).

### Как это устроено (простыми словами)

Раньше приложение и API были как комната без замка: кто знает адрес (например, твой IP и порт) — мог зайти, менять настройки, запускать генерацию, смотреть и удалять данные. Проверки «кто ты» не было.

Сейчас сделано так:

1. **Перед приложением поставили «охранника» (nginx).** Все запросы — и страница в браузере, и вызовы API, и WebSocket — сначала попадают в nginx. Само приложение (контейнер `app`) снаружи вообще не видно: к нему нельзя обратиться напрямую с твоего компьютера, только nginx изнутри Docker к нему стучится. То есть «дверь» одна — через nginx на порту 8080.

2. **Охранник спрашивает логин и пароль.** Пока не введёшь правильную пару (по умолчанию admin / admin), nginx не пропустит запрос к приложению. Неважно, открыл ли ты страницу в браузере, дергаешь API скриптом или подключаешься по WebSocket — везде сначала проверка. Пароли лежат в файле `nginx/.htpasswd`.

3. **Код приложения не трогали.** Проверку делает только nginx по своему конфигу и файлу с паролями. Go и React работают как раньше, просто до них запросы доходят только после «охранника».

Итог: зайти в приложение или к API может только тот, кто знает логин и пароль. Без этого — до сервера не добраться.

### Что сделано (кратко)

- Весь трафик идёт через nginx; приложение снаружи не открыто (нет проброса порта у контейнера `app`).
- Nginx перед каждым запросом проверяет логин и пароль (Basic Auth), пароли в `nginx/.htpasswd` (формат apr1).
- Защита включена без изменений в коде Go/React — только настройки nginx и файл паролей.

## Смена пароля или добавление пользователя

Сгенерируй новый `.htpasswd` (формат apr1 для nginx:alpine) и замени файл в проекте:

```bash
docker run --rm httpd:alpine htpasswd -nb admin ВАШ_ПАРОЛЬ > nginx/.htpasswd
```

Подставь нужное имя пользователя и пароль вместо `admin` и `ВАШ_ПАРОЛЬ`. Чтобы добавить второго пользователя, допиши в файл строку (или сгенерируй заново с двумя пользователями).

Затем перезапусти nginx:

```bash
docker compose restart nginx
```

**Важно:** не коммить в репозиторий файл `nginx/.htpasswd` с реальными паролями. Добавь его в `.gitignore` или храни пароли в секретах (переменные окружения, секретное хранилище при деплое).

## Запуск без Docker

При запуске **без Docker** (локальная разработка: Go + Node + PostgreSQL на своей машине) API и фронтенд **не защищены**: проверки логина и пароля нет. Используй такой режим только в доверенной среде (localhost, внутренняя сеть).

## Возможные улучшения безопасности

Ниже — что можно и нужно сделать для усиления защиты, особенно при доступе из интернета или в продакшене.

### Обязательно

- **Сменить пароль по умолчанию** — логин `admin` / пароль `admin` подходят только для локального теста. В продакшене задай свой пароль (см. раздел «Смена пароля» выше) и не храни обновлённый `nginx/.htpasswd` с реальными паролями в репозитории.

### Рекомендуется

- **HTTPS:** по умолчанию трафик идёт по HTTP. Для доступа из интернета или по недоверенным сетям подними nginx с TLS (сертификат, например Let's Encrypt) и редиректом HTTP → HTTPS.
- **Ограничение доступа по сети:** не открывай порт приложения в интернет без необходимости. Используй firewall, VPN или доступ только из доверенной подсети.

### Опционально

- **Защита в приложении:** при запуске без Docker (или при другом способе деплоя без nginx) API и фронт остаются без проверки. При необходимости можно добавить в бэкенд проверку API-ключа в заголовке или JWT — тогда защита будет работать и без nginx (единый механизм для разных вариантов деплоя).
