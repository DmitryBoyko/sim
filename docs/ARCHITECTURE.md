# Архитектура приложения: симуляция и детекция рейсов

## Обзор

Серверно-клиентское приложение для генерации имитационных данных (скорость и вес самосвала), векторизации и распознавания рейсов в реальном времени. Без авторизации.

## Стек

| Слой | Технологии |
|------|------------|
| Backend | Go 1.22+, Gin, pgx, Viper, goroutines, channels |
| Транспорт | REST API + WebSocket (стриминг данных в UI) |
| БД | PostgreSQL |
| Frontend | React, ECharts, WebSocket |

## Потоки данных

1. **Генерация**  
   Сервис `generator` в отдельной goroutine по таймеру (каждые R сек) вычисляет скорость и вес по текущей фазе рейса, накладывает шум и отправляет точку в канал.

2. **Очередь**  
   Сервис `queue` читает из канала, буферизует, отдаёт данные подписчикам (WebSocket) и в сервис распознавания. Очередь ограничена по времени (храним последние P минут).

3. **Распознавание**  
   Сервис `recognition` накапливает скользящее окно (по размерам шаблонов), строит вектор по выбранному режиму (Min-Max или Z-нормализация из настроек), сравнивает с соответствующим вектором шаблона (vector или zvector) по косинусному сходству в %. При совпадении ≥ порога и прохождении проверок «у оси» и охлаждения рейс сохраняется в БД, в UI уходит событие.

4. **UI**  
   Подключается по WebSocket, получает поток точек и события «рейс найден». REST — для настроек, шаблонов, истории, старт/стоп/очистка.

## Логика генерации по фазам

- **Погрузка** (T_load): скорость ≤ Vmin, вес линейно от 0 до M.
- **Перевозка** (T_transport): скорость в [Vmin, Vmax], вес в [Mmin, Mmax] + шум.
- **Разгрузка** (T_unload): скорость падает до Vmin, вес падает до 0.
- **Возврат** (T_return): скорость в [Vmin, Vmax], вес ≈ M_empty + шум.

Цикл повторяется бесконечно. На каждую величину накладывается шум (нормальный/равномерный), уровень задаётся в настройках.

## Векторизация и распознавание

- **Режим нормализации** задаётся в настройках (`recognition.use_z_normalization`): **Min-Max** (по умолчанию) или **Z-нормализация**. От него зависит, какой вектор шаблона и какая нормализация окна используются.
- **Шаблон**: при сохранении вычисляются **два** вектора — **vector** (Min-Max: нормализация ряда в 0…100 по min/max среза) и **zvector** (Z-нормализация: μ=0, σ=1 по срезу). Оба — конкатенация нормализованных скоростей и весов; длина = speed_count + weight_count. В БД хранятся оба; при смене режима в настройках используется соответствующий. Для старых шаблонов без `zvector` при первом сохранении настроек с включённой Z-нормализацией значения пересчитываются из сырых рядов.
- **Окно**: последние N скоростей и M весов (N, M по размеру текущего шаблона). При каждой новой точке — сдвиг sliding window; вектор окна строится тем же способом, что и выбранный вектор шаблона (Min-Max или Z).
- **Сравнение**: **косинусное сходство** между векторами, переведённое в проценты: (cos+1)/2·100. Одна метрика для обоих режимов. Порог в % настраиваемый; при совпадении с одним шаблоном и прохождении проверок «у оси» и охлаждения рейс сохраняется.

## API (кратко)

- `GET/PUT /api/settings` — параметры приложения (фазы, шумы, пороги, интервалы, распознавание в т.ч. use_z_normalization).
- `POST /api/control/start`, `stop`, `clear` — управление генерацией.
- `GET /api/data/operational` — последние P минут (при необходимости).
- `WS /ws` — стрим: новые точки `{t, speed, weight, phase?}` и события `{type: "trip_found", payload: {...}}`.
- `POST /api/templates` — сохранить выделенный участок как шаблон (тело: { from_ts, to_ts } или массив точек).
- `GET /api/templates` — список шаблонов.
- `GET /api/trips` — найденные рейсы (с пагинацией).
- `GET /api/history` — история данных за период; рейсы на графике (прямоугольники с подфазами).
- `GET /api/recognition/analysis` — состояние анализа для UI (шаблоны, окно, лучшее совпадение, **normalization_mode**: "min-max" | "z-norm").

Детали — в `docs/API.md`.

## База данных (основные таблицы)

- **app_params** — ключ/значение (JSON или отдельные поля) для всех настроек.
- **operational_data** — потоковые точки (ts, speed, weight, phase_id).
- **trip_templates** — шаблон рейса (id, name, created_at, speed_count, weight_count, raw_speed_json, raw_weight_json).
- **trip_template_vectors** — векторы шаблона (template_id, vector JSONB, zvector JSONB). **vector** — Min-Max нормализация; **zvector** — Z-нормализация (μ=0, σ=1). Оба вычисляются при сохранении шаблона.
- **detected_trips** — найденные рейсы (id, started_at, ended_at, template_id, match_percent, phases_json).
- **data_history** — снимки/архив данных за периоды (опционально, или выборка из operational_data за период).

Схемы детально — в `migrations/`.
